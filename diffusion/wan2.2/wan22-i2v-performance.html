Introduction

Wan2.2 is a Diffusion Transformer (DiT) family for text-to-video, image-to-video, and related modalities. The 14B I2V model stacks a dual-transformer MoE cascade — low-noise and high-noise experts — behind a VAE and text encoder. Serving it online means paying for every stage on every request: preprocess, encode, denoise, decode, and API overhead.

vLLM-Omni shipped Wan2.2 support across several releases. This post tracks image-to-video latency from v0.16.0 through v0.20.0, using the same workload shape on fixed hardware, and explains which optimizations moved the needle — not just which PRs merged.

Cookbook reference: Quick tables and serve commands live in the WAN2.2 index. This page is the narrative version for readers and blog publication.

The standardized benchmark

Starting in v0.20.0, upstream added a fixed perf JSON — test_wan22_i2v_vllm_omni.json (#3063) — and a nightly job on 2× H100. Three cases cover single-GPU and multi-GPU serving:

ConfigWorkloadCI baseline (H100)
Single device832×480, 81 frames, 4 steps26.0 s
USP2 + VAE-pp2 + HSDP + slicing832×480, 81 frames, 4 steps21.6 s
USP2 + VAE-pp2 + HSDP + slicing1280×720, 121 frames, 4 steps101.6 s

Those baselines are static assertion thresholds — nightly CI compares against them but does not rewrite them each release. For release-to-release deltas we ran a retro suite on 4× NVIDIA H200 (L20X SKU) with the same resolutions, step counts, seed, and concurrency.

One request end-to-end

Every benchmark request hits the same pipeline via POST /v1/videos:

Client→ Text encode→ Image preprocess→ VAE encode→ DiT ×4 steps→ VAE decode→ Video

On the standard USP2 config, DiT denoise dominates at 480p; at 720p with 121 frames, VAE and preprocess share more of the wall clock. Enable --enable-diffusion-pipeline-profiler and read stage lines in server logs before optimizing — fuse the largest box first.

How we optimized, step by step

1. Build the parallel stack (v0.14 – v0.16)

Before micro-kernels, Wan2.2 needed multi-GPU serving primitives: Ulysses sequence parallel, CFG parallel, VAE patch parallel, HSDP, and the OpenAI-style /v1/videos API. The perf test config test_wan22_i2v_usp2_vae_patch2_hsdp_slicing is the “fully wired” 4-GPU recipe from this phase.

2. Cut serving overhead (v0.18.0)

#1715 reduced IPC between online Wan2.2 processes — 37.5 s → 31.0 s (−17.5%) on a separate workload. #1504 sped cold start via multi-thread safetensors load. v0.18.0 is the serving-efficiency release; our H200 v0.18 retro numbers already include this step.

3. Shrink preprocess and VAE (v0.20.0)

PRChangeWhy it matters at 720p
#2963Remove duplicate video preprocessFewer copies before VAE encode
#2852Free GPU during I2V preprocessLess idle GPU blocking
#2391VAE FP32 → BF16Faster encode/decode, lower memory

H200 signal: 720p 93.67 s → 79.19 s combines these with DiT work.

4. Fuse the DiT loop (v0.20.0)

Wan2.2 14B I2V repeats RMSNorm, AdaLayerNorm, RoPE, and attention every step — ideal fusion targets:

  • #2583 / #2585 — fused RMSNorm and AdaLayerNorm (GPU)
  • #2393 — faster rotary embedding
  • #2459 — skip Ulysses SP when cross-attn sequence is short
  • #3327 — Flash Attention / CUBLAS path fix

H200 signal: USP2 @ 480p 20.26 s → 16.43 s (−18.9%) — largest single release jump.

5. Rebase runtime and lock CI (v0.20.0)

#3232 rebased to upstream vLLM v0.20.0. Part of the v0.18→v0.20 delta is the vLLM stack itself, not only omni-side Wan patches. #3063 added the nightly perf JSON so regressions surface on H100.

Checklist for your next change

  1. Reuse test_wan22_i2v_vllm_omni.json — do not invent one-off resolutions.
  2. Profile one request with --enable-diffusion-pipeline-profiler.
  3. Classify the bottleneck: preprocess, VAE, DiT, or API/IPC.
  4. Land one change; re-run on fixed hardware (same GPUs, same serve_args).
  5. Record latency_mean, throughput_qps, peak_memory_mb_mean.
  6. Compare: delta = (t_new - t_old) / t_old.

What’s next (v0.21+): pipeline parallel (#2322), GPU FP8 (#2920), VAE tiling (#3111) — extend the same table after each lands.

Reproduce on your hardware

v0.20.0+ (pytest + upstream JSON)

cd /path/to/vllm-omni
export CUDA_VISIBLE_DEVICES=0,1,2,3
export DIFFUSION_ATTENTION_BACKEND=FLASH_ATTN
export VLLM_WORKER_MULTIPROC_METHOD=spawn

pytest -s tests/dfx/perf/scripts/run_diffusion_benchmark.py \
  --test-config-file tests/dfx/perf/tests/test_wan22_i2v_vllm_omni.json

Retro A/B (v0.16 / v0.18 / v0.20 tags)

Configs: vllm-omni/benchmark_results/wan22_retro/config/. Default num-prompts=3, warmup-requests=0.

# v0.20
bash benchmark_results/wan22_retro/v0.20.0/run_benchmark.sh

# v0.18 (run_retro.py — i2v needs /v1/videos backend)
bash benchmark_results/wan22_retro/v0.18.0/run_benchmark.sh

# v0.16 (sync /v1/videos API; cleanup.sh if GPU workers linger)
bash benchmark_results/wan22_retro/v0.16.0/run_benchmark.sh

Standard serve (matches perf JSON)

export CUDA_VISIBLE_DEVICES=0,1,2,3
export DIFFUSION_ATTENTION_BACKEND=FLASH_ATTN
export VLLM_WORKER_MULTIPROC_METHOD=spawn

vllm serve Wan-AI/Wan2.2-I2V-A14B-Diffusers --omni \
  --usp 2 --vae-patch-parallel-size 2 --use-hsdp --vae-use-slicing \
  --enable-diffusion-pipeline-profiler
